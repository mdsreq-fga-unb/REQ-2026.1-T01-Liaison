## ADR-001: User com UUID PK e login por email

**Date:** 2026 | **Status:** Accepted

### Context
Identificador exposto na API (`/users/{id}/`) e relações entre apps. Login do produto é por email, não username.

### Decision
`User(AbstractUser)` com `id = UUIDField` PK e `USERNAME_FIELD="email"` (unique). `REQUIRED_FIELDS=["username"]`.

### Rationale
UUID evita enumeração sequencial de IDs na API. Email como credencial alinha ao fluxo de produto.

### Trade-offs
- ✅ IDs não enumeráveis; email é credencial natural.
- ⚠️ Org loga por CNPJ → precisa resolver CNPJ→email no `CustomTokenObtainPairView`.

## ADR-002: Campos de domínio movidos do User para Profiles dedicados

**Date:** 2026 | **Status:** Accepted

### Context
Estudante e organização têm campos distintos (matricula vs cnpj/razao_social). Inicialmente no User.

### Decision
`StudentProfile` e `OrganizationProfile` (1:1 com User). Campos `matricula`/`cnpj`/`endereco`/`telefone` removidos do User.

### Rationale
User magro e genérico; campos específicos de role no profile correspondente.

### Trade-offs
- ✅ Separação limpa por role; validações isoladas.
- ⚠️ Remover campo do User exigiu atualizar serializers + tests + tipos TS do frontend.

## ADR-003: JWT stateless sem refresh automático no client

**Date:** 2026 | **Status:** Accepted

### Context
API REST consumida por app mobile. Necessidade de auth sem sessão server-side.

### Decision
simplejwt HS256, access 30min / refresh 1d, `ROTATE_REFRESH_TOKENS=False`. Client guarda tokens em `SecureStore`; **sem** interceptor de refresh automático.

### Rationale
Stateless escala melhor; SecureStore persiste sessão após reload. Refresh automático adiado (gap conhecido).

### Trade-offs
- ✅ Sem estado de sessão no servidor; sessão sobrevive a F5/reload.
- ⚠️ Access expira em 30min sem renovação transparente → usuário pode precisar relogar.

### Alternatives Considered
- **Session cookies:** rejeitado — pior fit para mobile/stateless.

## ADR-004: Moderação de organização enforced no backend

**Date:** 2026 | **Status:** Accepted

### Context
RF06: orgs precisam de aprovação admin antes de operar. Frontend poderia esconder UI, mas é burlável.

### Decision
`OrganizationProfile.status` (pending/approved/rejected). `AdminOrganizationViewSet` (`IsAdmin`) faz approve/reject/request_info, gravando `AdminActionLog`. Enforce nas permissions/queries do DRF.

### Rationale
Segurança não pode depender do cliente. Log de auditoria para ações administrativas.

### Trade-offs
- ✅ Regra de negócio inviolável; trilha de auditoria.
- ⚠️ Mais lógica de permissão no backend.

## ADR-005: Storage condicional local/S3 via flag

**Date:** 2026 | **Status:** Accepted

### Context
Uploads de imagem (avatars, banners, galerias, fotos de vaga). Dev local não deve depender de S3.

### Decision
`USE_S3` (default False). Se True → `django-storages` + boto3 (bucket `liaison-media-2026`, us-east-2). Senão → `MEDIA_ROOT` local, servido por `static()` só com `DEBUG=True`.

### Rationale
Dev sem credenciais AWS; prod usa S3 durável.

### Trade-offs
- ✅ Dev simples; prod escalável.
- ⚠️ Dois caminhos de storage para testar.

## ADR-006: StyleSheet nativo + tokens, sem NativeWind

**Date:** 2026 | **Status:** Accepted

### Context
Estilização do app React Native.

### Decision
`StyleSheet` nativo com tokens centralizados em `src/theme/` (colors/typography/spacing). Sem NativeWind/Tailwind.

### Rationale
Menos dependências; controle direto; tokens garantem consistência.

### Trade-offs
- ✅ Zero camada extra de build de CSS.
- ⚠️ Chave de StyleSheet errada = `undefined` silencioso (testes passam). Disciplina de token manual.
